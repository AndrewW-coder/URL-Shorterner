--- Phase 2: Building Lambda Function and Dynamo DB Table ---

- DynamoDB
    - this is AWS's NoSQL database
        - basically a giant dictionary: given short code you get back long url
    - we can create using:

    - aws dynamodb create-table ^
      --table-name urls ^
      --attribute-definitions AttributeName=short_code,AttributeType=S ^
      --key-schema AttributeName=short_code,KeyType=HASH ^
      --billing-mode PAY_PER_REQUEST ^
      --region us-east-1

    - where:
        --table-name urls -> table is called "urls"
        --attribute-definitions AttributeName=short_code,AttributeType=S -> our key is a string
        --key-schema AttributeName=short_code,KeyType=HASH its the primary key

    - put_item
    - get_item
    - short_code: primary key -> lookups are instant no matter how many URLs are stored

--- Phase 2: Deploying Functions to AWS ---

Creating an IAM Role for Lambda

    aws iam create-role --role-name lambda-url-shortener-role `
        --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"lambda.amazonaws.com"},"Action":"sts:AssumeRole"}]}'

    - We need to give the Lambda function permissions to interact with DynamoDB

    aws iam attach-role-policy --role-name lambda-url-shortener-role `
        --policy-arn arn:aws:iam::aws:policy/AmazonDynamoDBFullAccess

    - Now we have given the badge to the Lambda function and we attatch permission to alter DynamoDB

    aws iam attach-role-policy --role-name lambda-url-shortener-role `
        --policy-arn arn:aws:iam::aws:policy/CloudWatchLogsFullAccess

    - Allows the function to write logs, CloudWatch is AWS's logging service, we can see error msgs

--- Phase 2: Zipping and Deploying ---

    Compress-Archive -Path lambda_function.py -DestinationPath function.zip

    - we need to zip lambda file since Lambda doesn't accept raw .py files

    aws lambda create-function `
        --function-name shorten_url ` -> names the function in AWS
        --runtime python3.11 ` -> language engine
        --role arn:aws:iam::YOUR_ACCOUNT_ID:role/lambda-url-shortener-role ` -> attaches permissions in first block
        --handler lambda_function.lambda_handler ` -> tells Lambda which function to call when triggered
        --zip-file fileb://function.zip ` -> read this as a binary file from disk (fileb)
        --region us-east-1 -> region

    - main deployment function 
    
    aws lambda invoke ` -> manually triggers lambda
        --function-name shorten_url `
        --payload '{"body": "{\"long_url\": \"https://google.com\"}"}' `
        --cli-binary-format raw-in-base64-out ` -> we need to tell AWS CLI that we are entering JSON
        output.json

    - testing, we trigger the function without needing an HTTP request

- all permissions need to be explicitly granted in AWS
- IAM role is how we grant Lambda permission to run DynamoDB
- --handler is how AWS knows which exact function to run
- fileb:// is how we get binary files up to the cloud

- boto3 is python library that lets us connect to AWS service
    - eg: DynamoDB, S3
    - installed automatically inside Lambda

- lambda_handler(event, context) is the entry point that AWS always calls, kinda like main()
    - event contains whatever the caller send (the URL here)
    - context contains metadata about Lambda environment

--- Phase 3: Creating the API ---
    
Creating the API

    aws apigatewayv2 create-api ` -> modern version of API
        --name url-shortener-api ` -> names the API
        --protocol-type HTTP ` -> specifies HTTP API
        --region us-east-1

    - HTTP API is newer and cheaper than REST API in API Gateway
    - HTTP API is better since its simpler to configure and has lower latency
    - this command builds the restaurant, still no menus, staff, tables, etc

Giving API Gateway permission to invoke Lambda
    
    - AWS operates on a principle called least privilege
        - by default, every service is completely locked off from every other service
    - we need to give API Gateway permission to trigger Lambda functions

    aws lambda add-permission `
        --function-name shorten_url ` <- we replace shorten_url with redirect_url to giv perm to that
        --statement-id apigateway-shorten ` -> unique name for permission rule
        --action lambda:InvokeFunction ` -> specifically allows to run this function
        --principal apigateway.amazonaws.com ` -> only API Gateway can use this permission
        --region us-east-1

Creating Integrations

    - integrations are the wiring between API Gateway and Lambda function
    - says "when something needs to go to lambda, heres exactly which lambda and how to talk to it"
    - You create one integration per Lambda function, then attach routes to them

    - getting info of each function
    
    aws lambda get-function `
        --function-name shorten_url `
        --query 'Configuration.FunctionArn' `
        --output text

    - the integration-uri is the ARN of the Lambda function, unique global address in AWS so API Gateway knows which func to call

    - creating integration for each funnction

    aws apigatewayv2 create-integration `
        --api-id $API_ID `
        --integration-type AWS_PROXY `
        --integration-uri YOUR_SHORTEN_URL_FUNCTION_ARN `
        --payload-format-version 2.0 `
        --region us-east-1

    - AWS_PROXY means that API Gateway passes the entire HTTP request to Lambda
    - it returns whatever Lambda responds
        - this is why our lambda_handler reads from event['body'] and event['pathParameters']

Creating routes
    
    - routes map HTTP methods + URL paths to integrations
    - define what URLs exist and what HTTP methods they accept

    - We need two routes:
        - POST /shorten to create short URLs -> when someone sends POST request to /shorten path, route it to shorten_url
        - GET /{code} to redirect -> when someone sends a GET request to any single-segment pass, route to redict_url
            - {code} is a path parameter, matches anything like /abc123, etc

    - Route: POST /shorten
    aws apigatewayv2 create-route `
        --api-id $API_ID `
        --route-key "POST /shorten" ` -> accepts long url, returns short code
        --target integrations/$SHORTEN_INTEGRATION_ID `
        --region us-east-1

    - Route: GET /{code}
    aws apigatewayv2 create-route `
        --api-id $API_ID `
        --route-key "GET /{code}" ` -> any GET to /abc123 triggers redirect_url, {code} becomes event['pathParameters']['code'] in Lambda
        --target integrations/$REDIRECT_INTEGRATION_ID ` -> Points to the integration from previous step
        --region us-east-1

Deploying and Testing
    
    - we need to create a 'stage', kind of like a published snapshot and then test with real HTTP request
    - staging is like the act of publishing

    aws apigatewayv2 create-stage `
        --api-id $API_ID `
        --stage-name prod `
        --auto-deploy `
        --region us-east-1

    Invoke-WebRequest -Uri "$API_URL/shorten" ` -> sends real HTTP request from machine to live API
        -Method POST ` -> sets HTTP method
        -ContentType "application/json" ` -> tells server what format body is in
        -Body $payload | Select-Object -ExpandProperty Content -> puts response obj into ailfter that extracts just body text

- the chain is: internet -> API Gateway -> Lambda -> DynamoDB
- the API Gateway's job is to receive HTTP requests, check which route matches, then hand of request to right Lambda

--- Phase 4: Event Log and Analytics ---

Creating a clicks table in DynamoDB

    aws dynamodb create-table `
    --table-name clicks `
    --attribute-definitions `
        AttributeName=click_id,AttributeType=S ` -> diff from SQL, only need to define primary key (click_id), type String
    --key-schema `
        AttributeName=click_id,KeyType=HASH `  -> click_id is the partition key = main identifier to locate rows, constant time
    --billing-mode PAY_PER_REQUEST `
    --region us-east-1

    - we don't add clicks data to URL table since they are different shapes
        - the URL table has one row per short URL, updated in place each time smth changes
        - clicks table has one row per click event, only grows never updates
        - mixing these two would be awkward and expensive


    - each recorded click looks like:
    {
        "click_id":   "a3f9c2b1-...",   # unique ID per click (UUID)
        "short_code": "xK9mPq",         # which short URL was clicked
        "timestamp":  "2024-04-14T10:23:00Z",
        "country":    "United States",
        "city":       "Seattle",
        "device":     "Mobile",
        "browser":    "Chrome"
    }

    - this table design is called an 'event log'
        - recording every event that happens instead of just updating a total

Geo Lookup Layer

    - to find visitors country and city from IP address, we can use ip-api.com
    - Lambda will call this during each redirect

Updated lambda_function in redirect_url

    - we are making an outbound network request from inside AWS to an external service on the internet
    - try/except is defensive programming, we don't let errors crash lambda and break redirect

Redeploying and Verifying Analytics

    Compress-Archive -Path lambda_function.py `
        -DestinationPath function.zip -Force

    aws lambda update-function-code `
        --function-name redirect_url `
        --zip-file fileb://function.zip `
        --region us-east-1

    cd ..

    - we are using update-function-code since the function already exists

    - we can see the clicks table with

    aws dynamodb scan `
        --table-name clicks `
        --region us-east-1

    - Query Clicks for Specific Short Code

    aws dynamodb query `
        --table-name clicks `
        --index-name short_code-index `
        --key-condition-expression "short_code = :sc" `
        --expression-attribute-values '{":sc": {"S": "xK9mPq"}}' `
        --region us-east-1

    - query above requires a GSI (Global Secondary Index) on short_code
    - more for production queries

- counter vs event log
    - counter: we just increment the click count by 1
    - event log: we know who clicked and when, recording everything

Concepts:
    - Serverless / FaaS : Lambda only runs on request, no idle cost and infinite scale
    - Event Driven Design : Each HTTP request triggers one unique function
    - Least Privilege IAM : every role only has the persmissions it actually needs
    - NoSQL key design : Partition key choice, O(1) lookups by primary key
    - Event sourcing : Storing events rather than updating state in place
    - Dual-write pattern : counter for fast reads, event log for deep queries
    - Zero-downtime deploy : update function code replaces code with no restaurant
    - CORS : Why browsers enforce it, what the browser doesn
    - S3 static hosting : File bucket as CDN - backed web server - no infra to manage
